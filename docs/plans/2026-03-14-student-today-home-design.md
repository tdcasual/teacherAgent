# Student Today-First Homepage Design

**Goal:** 在不改变现有后端契约、聊天能力和学生验证主流程的前提下，把学生端首屏从“进入聊天”重构为“进入今天任务”，让学生打开页面后 3 秒内明确知道今天要做什么、当前做到哪一步、下一步点哪里开始。

**Architecture:** 保持现有学生端状态管理、会话管理、聊天轮询、附件上传和移动端壳不变，仅重构前端首屏信息架构与状态表达。新增一个位于聊天执行态之前的 `Today-first` 首页层，围绕单一主任务容器组织 `待生成 / 生成中 / 未开始 / 进行中 / 已提交` 五种状态，再把聊天区降级为任务执行工具而不是默认首页。

**Tech Stack:** React 19 + TypeScript + Tailwind CSS 4 + Vite + Vitest + Playwright

---

## Product Intent

- 学生端下一阶段不是继续做“聊天更方便”，而是做“今天任务更明确”。
- 首页的职责不是展示系统能力，而是帮助学生快速完成判断：
  - 今天的任务是什么
  - 现在处于什么状态
  - 该从哪里开始或继续
- 由于作业可以自动生成，学生端不存在真正的“无任务”状态；最低承诺应当是“今天总有一条可执行主路径”。

## Design Direction

- Aesthetic: `专注清爽`
- Product metaphor: `学习桌面`，不是 IM 工具，不是 SaaS 后台
- Key memory point: 学生每次打开都能立刻看到 `今天先做什么`

### Visual principles

- 减少通用后台式白卡片堆叠，只保留一个视觉重心：`主任务卡`
- 使用安静的暖灰背景和克制的强调色，不做强 AI 感装饰
- 大标题、短文案、清晰分隔，避免次级入口与主任务竞争
- 历史会话、补充入口、自由提问一律降级

---

## Information Architecture

学生端首屏改为自上而下 5 个区域：

1. `今日标题区`
   - 今日日期
   - 课程名或任务上下文
   - 一句极短说明，如“今天先完成本次物理练习”

2. `主任务卡`
   - 今日任务标题
   - 截止时间 / 预计时长 / 老师要求摘要
   - 当前状态标签
   - 唯一主按钮

3. `准备材料区`
   - 题目附件
   - 老师说明
   - 参考资料
   - 只展示完成当前任务所需的最小入口

4. `学习进度区`
   - 用步骤或状态线展示当前阶段
   - 不用装饰图表，不做复杂数据面板

5. `历史与补充入口`
   - 历史任务
   - 全部会话
   - 自由提问
   - 放在首屏底部或折叠区

## Core State Model

首页围绕同一个 `主任务卡` 承载 5 种状态：

### 1. 待生成

- 含义：今天任务存在，但内容尚未生成
- 主标题：`今日任务尚未生成`
- 说明文案：`系统会根据今天安排准备练习内容`
- 主按钮：`生成今日任务`

### 2. 生成中

- 含义：系统正在准备任务内容
- 主标题：`正在准备今天的任务`
- 反馈方式：展示简洁步骤反馈，例如：
  - 读取课程安排
  - 生成题目
  - 整理提交入口
- 主按钮：可禁用或转为 `稍后查看`

### 3. 未开始

- 含义：任务已就绪，但学生尚未开始
- 主标题：显示任务名称
- 元信息：截止时间、预计时长、老师要求摘要
- 主按钮：`开始今日任务`

### 4. 进行中

- 含义：任务已经开始但未提交
- 主标题：`继续今日任务`
- 元信息：最近学习时间、当前步骤、是否有未提交内容
- 主按钮：`继续完成`

### 5. 已提交

- 含义：今日任务已完成提交
- 主标题：`今天的任务已提交`
- 元信息：提交时间、老师反馈状态、是否允许补充提问
- 主按钮：`查看本次提交`

## Critical rule

- 不出现“今天没有任务”
- 无论任何状态，页面中心始终是同一个主任务容器
- 无论任何状态，只保留一个绝对主按钮

## Screen Structure

### Home state

- 默认进入 `Today-first` 首页，而不是聊天页
- 首页先承担判断与选择职责
- 只有点击主任务 CTA 后，才进入聊天/答题执行态

### Execution state

- 聊天区是任务执行工具
- 任务上下文、题目要求、已上传内容、提交动作围绕当前任务组织
- 从执行态返回首页时，应保留首页主卡状态同步

### Layout recommendation

- Desktop:
  - 上部为首屏信息区
  - 中部为主任务卡
  - 下部为材料、进度、历史入口
- Mobile:
  - 保持移动端壳
  - 首屏仍先展示主任务卡
  - 聊天页作为后续进入的执行页，而不是 tab 默认落点

## Component Plan

建议新增或重构以下组件：

- `StudentTodayHome`
  - 学生首页外层容器
- `TodayHero`
  - 日期、课程、说明文案
- `TodayTaskCard`
  - 统一承载五种状态与唯一主 CTA
- `TaskMaterialList`
  - 材料、附件、说明入口
- `LearningProgressRail`
  - 任务当前步骤和阶段
- `StudentHistoryEntry`
  - 历史任务、全部会话、自由提问入口

建议保留并重接：

- `ChatPanel`
- `SessionSidebar`
- `StudentTopbar`
- 现有学生验证、聊天轮询、附件上传、会话状态逻辑

---

## UX Writing Rules

- 从“系统术语”改为“任务术语”
- 避免使用：
  - 会话
  - pending
  - verify
  - job
- 统一改成学生可以立刻理解的表达：
  - 今日任务
  - 开始今日任务
  - 继续完成
  - 已提交，等待老师反馈
  - 正在准备今天的练习

---

## Motion and Feedback

- 动效只服务状态切换，不做装饰
- 重点动效场景：
  - 首页首次进入时主任务卡淡入
  - `待生成 -> 生成中 -> 未开始`
  - `未开始 -> 进行中`
  - `进行中 -> 已提交`
- 动画用 `opacity + transform`
- 避免高度抖动和复杂弹跳

## Implementation Phases

## Phase 1: Homepage skeleton

- 新建学生端首页骨架
- 完成五种主状态的静态展示
- 保证桌面端与移动端都能稳定显示

## Phase 2: State wiring

- 将首页状态映射到现有学生任务/聊天状态
- 接通 `开始今日任务 / 继续完成 / 查看本次提交`
- 让首页和聊天执行态共享同一任务上下文

## Phase 3: Visual refinement

- 拆分学生端专属语义 token
- 降低现有通用后台感
- 统一按钮层级、状态标签、空态、材料列表视觉

## Phase 4: Validation

- 增加组件测试覆盖五种首页状态
- 增加 E2E 验证：
  - 默认进入首页
  - 主按钮随状态变化
  - 进入执行态后上下文正确
  - 已提交状态不再误导继续作答

---

## Success Criteria

- 学生首次进入后 3 秒内能说出今天应该做什么
- 首屏主按钮只有一个，且始终与当前状态一致
- 聊天页不再承担首页职责
- 学生端与老师端首屏气质明显区分
- 不新增后端 API 也能完成第一轮改版

## Non-goals

- 不在本轮重做学生聊天能力
- 不在本轮引入新的学习数据图表系统
- 不在本轮扩展新的业务状态机
- 不为了设计感增加额外流程步骤
